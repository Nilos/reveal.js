Consensus Algorithms

Making distributed processes agree on values

by Nils Kenneweg

Motivation


  • State Machine Replication
  • De facto: Database Replication

Definitions



Non-Triviality: Only proposed values will be chosen.

Safety: At most one value will be chosen by the system.

Liveness: If a value has been proposed it is possible that a learner will learn some value.

Safety: Not two values can be learned by different learners.
Liveness: Those values do not have to be the same. If two values are proposed it might be that one is ignored completely.

Assumptions


Processors can fail but not loose state or lie.
Messages can be duplicated or lost but not be corrupted.

Paxos


by Leslie Lamport (1989, 2001)

Roles



Client: Issues a request to proposers

Proposer: Proposes value to acceptors.
Acceptor: Vote on which value is chosen
Learner: Receive chosen value and change their state

Leader: Designated Proposer who issues all proposals

Leader is only necessary for progress!

The algorithm


4 Phases

  1. Prepare(number)
  2. Promise(number, value)
  3. Accept!(number, value)
  4. Accepted(number, value)

Phase 1: Prepare


Proposer:

  • receives a request from a client.
  • increases his internal counter.

ProposerAcceptorLearner
---->
Prepare(counter)

Phase 2: Promise


Acceptor: If internal counter is smaller than counter given in prepare request:

ProposerAcceptorLearner
<----
Promise(counter)

Phase 3: Accept!


Proposer:
If a quorum of acceptors send a promise with my number or a smaller number
Change my value to the value with the smallest number

ProposerAcceptorLearner
---->
Accept!(counter, value)

Phase 4: Accepted


Acceptor: If my internal counter equals the given counter:

ProposerAcceptorLearner
<------------------------------------------------->
Accepted(counter, value)
Use value
Respond client request

Leader

  • Not necessary
  • Two proposers might steal each others promises otherwise
  • Ensures Liveness

Disadvantages and Optimizations


  • Accepted has to be sent from every acceptor to every learner
  • Multi-Paxos: Do only one Prepare and Promise and only repeat Accept and Accepted by adding instance numbers
  • Sent prepare and accept only to a quorum of acceptors

Raft


by Diego Ongaro and John Ousterhout (2013)

Idea and Motivation

Paxos is difficult. Make a simpler approach.
Try not to agree on one value but to agree on a list of actions
Have a strong leader
Distinguish terms by using a logical clock.

Roles


Leader: Receives requests and issues updates and heartbeats.
Follower: Replicates received entries. Does not issue messages.
Candidate: Wants to become leader. Issues RequestVote message.

Leader Election


If the old leader timed out

Increase internal term counter

Vote for myself, request vote from everone else.
Only grant votes if my state (term and highest index) is newer.
If I am accepted by a majority, I am the new leader!

Timeout intervals should be randomized on a certain interval because otherwise everyone will start to request votes at the same time and no one will win.
If voting fails retry with new term.

Leader Tasks


  • Notify others about leadership (regular heartbeat)
  • Handles all requests
  • Sends heartbeats on a regular basis
  • Updates clients by sending append entries message
  • Keeps track of which data has been synced to which client
Heartbeats and AppendEntry are the same!

State


Persistent

  • currentTerm: Logical clock. Is always set to max known term.
  • votedFor
  • entry log[]


Volatile

  • commitIndex: highest index known to be commited
  • lastApplied: highest index applied to state machine


Leader

  • nextIndex[]: entry to send next
  • matchIndex[]: entry known to be replicated
