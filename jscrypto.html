Security

Security

Javascript Cryptography Considered Harmful
Source: matasano.com (around 2011)

The problems

  • Re-installation every time one accesses the page
  • secure code transfer
  • secure key storage
  • Lack of random numbers
  • Runtime is changeable ("malleable")

secure code transfer

Source code might be changed in transit or on the server

  • Use SSL
  • Use HSTS
  • Use Application Cache to make updates clearer
  • Seperate code and data server
  • Allow local code execution (possible with SPA)

secure key storage


  • Key can only be stored unencrypted
  • No KeyChain access
  • Unable to protect memory (disallow write to disk)

Lack of random numbers


Solved

The getRandomValues method generates cryptographically random values.
Chrome 11, Firefox 21, IE 11, Opera 15, Safari 3.1

changeable runtime


  • eval and property based JS make review hard
  • changeable properties enable attackers to change everything

Solution


  • content-security-policy
    default-src 'self'
  • seperated application code and data server
  • Advanced: seperate security code into own iframe and use message passing

Best practices


  • Use SSL - if that solves your crypto-problem, JS-Crypto is not for you
  • Seperate code and data-server
  • Static code, no generated code

Ideas for higher security


  • Crypto-Code in Iframe
  • Secure JavaScript
  • CSP: Disable all external scripts and resources

The Future
