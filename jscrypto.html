Javascript Cryptography Considered Harmful
Source: matasano.com

The problems

  • Lack of random numbers
  • secure code transfer
  • Runtime is changeable ("malleable")
  • secure key storage
  • Re-installation every time one accesses the page

Lack of random numbers


Javascript has a random number generator in modern browsers
Older browsers can use mouse movements

secure code transfer

When downloading the source code via http, an attacker could easily change the crypto code

Use SSL and HSTS
If SSL solves your crypto problem, use it instead of JS-Crypto

changeable runtime


  • eval and property based JS make review hard
  • changeable properties enable attackers to change everything


  • content-security-policy

  • static application code eases review
  • Advanced: seperate security code into own iframe and use message passing

Use content-security-policy to disable inline JavaScript, eval and alike.
Seperate Application and Data (do not generate JS-Code on your server based on input)

secure key storage


This is not solveable at the moment
If a key is stored on the device, it can only be secured by the users password
Hibernation remains a problem

Auto re-install on page load

Every time one opens the web-page, the JS-Crypto-Application is reinstalled

Possible improvement: Use ApplicationCache

Best practices


  • Use SSL - if that solves your crypto-problem, JS-Crypto is not for you
  • Seperate code and data-server
  • Static code, no generated code

Ideas for higher security


  • Crypto-Code in Iframe
  • Secure JavaScript
  • CSP: Disable all external scripts and resources

Practical Examples


  • Simple encrypted chat
  • Dropbox-like Cloud Storage with encryption
  • Removing passwords on the server in favor of e.g. secure-remote-password

The Future


Standardization of Web-Crypto API

  • Faster encryption
  • Secure Key Storage
  • One single API

Recap


  • JS-Crypto is possible today but hard
  • Web-Crypto API will make it easier and more secure
  • Be sure to have a valid use case
    (saving on an SSL-certificate is not!)
